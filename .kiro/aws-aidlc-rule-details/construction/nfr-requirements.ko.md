# NFR Requirements

## 전제 조건
- unit에 대한 Functional Design이 완료되어야 함
- Unit functional design 아티팩트가 사용 가능해야 함
- 실행 계획에서 NFR Requirements 단계 실행이 표시되어야 함

## 개요
unit에 대한 비기능적 요구사항을 결정하고 기술 스택 선택을 수행.

## 실행 단계

### 1단계: Functional Design 분석
- `aidlc-docs/construction/{unit-name}/functional-design/`에서 functional design 아티팩트 읽기
- 비즈니스 로직 복잡성 및 요구사항 이해

### 2단계: NFR Requirements 계획 생성
- NFR 평가를 위한 체크박스 [] 포함 계획 생성
- 확장성, 성능, 가용성, 보안에 집중
- 각 단계에는 체크박스 []가 있어야 함

### 3단계: 상황에 맞는 질문 생성
**지침**: functional design을 철저히 분석하여 NFR 명확화가 시스템 품질과 아키텍처 결정을 개선할 수 있는 모든 영역을 식별. 포괄적인 NFR 커버리지를 보장하기 위해 적극적으로 질문하기.

**중요**: 시스템 품질에 영향을 줄 수 있는 모호성이나 누락된 세부사항이 있을 때는 기본적으로 질문하기. 잘못된 NFR 가정을 하는 것보다 너무 많은 질문을 하는 것이 낫습니다.

- [Answer]: 태그 형식을 사용하여 질문 포함
- 모든 모호성, 누락된 정보, 또는 명확화가 필요한 영역에 집중
- 사용자 입력이 NFR 및 기술 스택 결정을 개선할 수 있는 곳에서 질문 생성
- **의심스러울 때는 질문하기** - 과신은 시스템 품질 저하로 이어짐

**평가할 질문 카테고리** (모든 카테고리 고려):
- **확장성 요구사항** - 예상 부하, 성장 패턴, 확장 트리거, 용량 계획에 대해 질문
- **성능 요구사항** - 응답 시간, 처리량, 지연 시간, 성능 벤치마크에 대해 질문
- **가용성 요구사항** - 가동 시간 기대치, 재해 복구, 장애 조치, 비즈니스 연속성에 대해 질문
- **보안 요구사항** - 데이터 보호, 규정 준수, 인증, 권한 부여, 위협 모델에 대해 질문
- **기술 스택 선택** - 기술 선호도, 제약 조건, 기존 시스템, 통합 요구사항에 대해 질문
- **신뢰성 요구사항** - 오류 처리, 내결함성, 모니터링, 알림 요구사항에 대해 질문
- **유지보수성 요구사항** - 코드 품질, 문서화, 테스트, 운영 요구사항에 대해 질문
- **사용성 요구사항** - 사용자 경험, 접근성, 인터페이스 요구사항에 대해 질문

### 4단계: 계획 저장
- `aidlc-docs/construction/plans/{unit-name}-nfr-requirements-plan.md`로 저장
- 사용자 입력을 위한 모든 [Answer]: 태그 포함

### 5단계: 답변 수집 및 분석
- 사용자가 모든 [Answer]: 태그를 완료할 때까지 대기
- **필수**: 모든 응답을 주의 깊게 검토하여 모호하거나 애매한 답변 확인
- **중요**: 불분명한 응답에 대해서는 후속 질문 추가 - 모호성을 가지고 진행하지 않음
- "상황에 따라", "아마도", "확실하지 않음", "혼합", "중간 정도", "표준", "일반적인" 같은 응답 찾기
- 모호성이 감지되면 명확화 질문 파일 생성
- **모든 모호성이 해결될 때까지 진행하지 않음**

### 6단계: NFR Requirements 아티팩트 생성
- `aidlc-docs/construction/{unit-name}/nfr-requirements/nfr-requirements.md` 생성
- `aidlc-docs/construction/{unit-name}/nfr-requirements/tech-stack-decisions.md` 생성

### 7단계: 완료 메시지 제시
- 이 구조로 완료 메시지 제시:
     1. **완료 공지** (필수): 항상 이것으로 시작:

```markdown
# 📊 NFR Requirements 완료 - [unit-name]
```

     2. **AI 요약** (선택사항): NFR 요구사항의 구조화된 불릿 포인트 요약 제공
        - 형식: "NFR requirements 평가가 [설명]을 식별했습니다:"
        - 주요 확장성, 성능, 가용성 요구사항 나열 (불릿 포인트)
        - 식별된 보안 및 규정 준수 요구사항 나열
        - 기술 스택 결정 및 근거 언급
        - 워크플로우 지침 포함하지 않기 ("검토해 주세요", "알려주세요", "다음 단계로 진행", "진행하기 전에")
        - 사실적이고 내용 중심으로 유지
     3. **형식화된 워크플로우 메시지** (필수): 항상 이 정확한 형식으로 끝내기:

```markdown
> **📋 <u>**검토 필요:**</u>**  
> 다음 위치에서 NFR 요구사항을 검토해 주세요: `aidlc-docs/construction/[unit-name]/nfr-requirements/`



> **🚀 <u>**다음 단계는?**</u>**
>
> **다음을 수행할 수 있습니다:**
>
> 🔧 **변경 요청** - 검토를 바탕으로 NFR 요구사항 수정 요청  
> ✅ **다음 단계로 계속** - NFR 요구사항을 승인하고 **[next-stage-name]**으로 진행

---
```

### 8단계: 명시적 승인 대기
- 사용자가 NFR 요구사항을 명시적으로 승인할 때까지 진행하지 않음
- 승인은 명확하고 애매하지 않아야 함
- 사용자가 변경을 요청하면 요구사항을 업데이트하고 승인 프로세스 반복

### 9단계: 승인 기록 및 진행 상황 업데이트
- 타임스탬프와 함께 audit.md에 승인 로그
- 타임스탬프와 함께 사용자의 승인 응답 기록
- aidlc-state.md에서 NFR Requirements 단계를 완료로 표시
