# User Stories - 상세 단계

## 목적
**요구사항을 사용자 중심의 스토리와 수락 기준으로 변환**

User Stories는 다음에 중점을 둡니다:
- 비즈니스 요구사항을 사용자 중심의 내러티브로 변환
- 각 스토리에 대한 명확한 수락 기준 정의
- 다양한 이해관계자 유형을 나타내는 사용자 페르소나 생성
- 팀 간 공통 이해 구축
- 구현을 위한 테스트 가능한 명세 제공

## 전제 조건
- Workspace Detection이 완료되어야 함
- Requirements Analysis 권장 (사용 가능한 경우 요구사항 참조 가능)
- Workflow Planning에서 User Stories 단계 실행을 지시해야 함

## 지능형 평가 가이드라인

**USER STORIES 실행 시기**: 진행하기 전에 이 향상된 평가를 사용하세요:

### 높은 우선순위 실행 (항상 실행)
- **새로운 사용자 기능**: 사용자가 직접 상호작용할 새로운 기능
- **사용자 경험 변경**: 기존 사용자 워크플로우나 인터페이스 수정
- **다중 페르소나 시스템**: 다양한 유형의 사용자에게 서비스하는 애플리케이션
- **고객 대면 API**: 외부 사용자나 시스템이 사용할 서비스
- **복잡한 비즈니스 로직**: 여러 시나리오나 비즈니스 규칙이 있는 요구사항
- **교차 팀 프로젝트**: 여러 팀 간 공통 이해가 필요한 작업

### 중간 우선순위 실행 (복잡성 평가)
- **Backend 사용자 영향**: 사용자 경험에 간접적으로 영향을 미치는 내부 변경
- **성능 개선**: 사용자에게 보이는 이점이 있는 향상
- **통합 작업**: 사용자 워크플로우에 영향을 미치는 시스템 연결
- **데이터 변경**: 사용자 데이터, 보고서 또는 분석에 영향을 미치는 수정
- **보안 강화**: 사용자 인증이나 권한에 영향을 미치는 변경

### 복잡성 평가 요소
중간 우선순위의 경우, 다음 중 하나라도 해당하면 user stories를 실행:
- **범위**: 변경이 여러 컴포넌트나 사용자 접점에 걸쳐 있음
- **모호성**: 스토리로 명확히 할 수 있는 불분명한 측면이 있는 요구사항
- **위험**: 높은 비즈니스 영향이나 오해 가능성
- **이해관계자**: 요구사항에 여러 비즈니스 이해관계자 참여
- **테스팅**: 사용자 수락 테스트가 필요함
- **옵션**: 여러 유효한 구현 접근법 존재

### 단순한 경우에만 건너뛰기
- **순수 리팩토링**: 사용자 영향이 전혀 없는 내부 코드 개선
- **격리된 버그 수정**: 명확한 범위의 단순하고 잘 정의된 수정
- **인프라만**: 사용자 대면 효과가 없는 변경
- **개발자 도구**: 빌드 프로세스, CI/CD 또는 개발 환경 변경
- **문서화**: 기능에 영향을 주지 않는 업데이트

### 기본 결정 규칙
**의심스러울 때는 user stories를 포함하고 명확화 질문을 하세요.** 적절한 명확화와 함께 포괄적인 스토리를 만드는 오버헤드는 일반적으로 다음 이점으로 상쇄됩니다:
- 더 명확한 요구사항 이해
- 더 나은 팀 정렬
- 개선된 테스트 기준
- 향상된 이해관계자 커뮤니케이션
- 구현 위험 감소
- 개발 중 비용이 많이 드는 변경 감소
- 더 나은 사용자 경험 결과

# PART 1: 계획

## 단계 1: User Stories 필요성 검증 (필수)

**중요**: user stories를 진행하기 전에 이 평가를 수행하세요:

### 평가 프로세스
1. **요청 컨텍스트 분석**:
   - 원래 사용자 요청과 요구사항 검토
   - 사용자 대면 vs 내부 전용 변경 식별
   - 작업의 복잡성과 범위 평가
   - 비즈니스 이해관계자 참여 평가

2. **평가 기준 적용**:
   - 높은 우선순위 지표 확인 (항상 실행)
   - 중간 우선순위 요소 평가 (복잡성 기반 결정)
   - 건너뛰어야 할 단순한 경우가 아님을 확인

3. **평가 결정 문서화**:
   - `aidlc-docs/inception/plans/user-stories-assessment.md` 생성
   - 이 요청에 대해 user stories가 가치 있는 이유 포함
   - 적용되는 특정 평가 기준 참조
   - 예상 이점 설명 (명확성, 테스팅, 이해관계자 정렬)

4. **정당화된 경우에만 진행**:
   - User stories는 프로젝트에 명확한 가치를 추가해야 함
   - 평가는 구체적인 이점이 오버헤드를 상회함을 보여야 함
   - 결정은 프로젝트 이해관계자에게 방어 가능해야 함

### 평가 문서화 템플릿
```markdown
# User Stories 평가

## 요청 분석
- **원래 요청**: [간략한 요약]
- **사용자 영향**: [직접/간접/없음]
- **복잡성 수준**: [단순/중간/복잡]
- **이해관계자**: [관련 당사자 목록]

## 충족된 평가 기준
- [ ] 높은 우선순위: [적용 가능한 기준 목록]
- [ ] 중간 우선순위: [복잡성 정당화와 함께 적용 가능한 기준 목록]
- [ ] 이점: [user stories에서 예상되는 가치]

## 결정
**User Stories 실행**: [예/아니오]
**근거**: [상세한 정당화]

## 예상 결과
- [user stories가 제공할 구체적인 이점 목록]
- [스토리가 프로젝트 성공을 개선하는 방법]
```

## 단계 3: 컨텍스트에 적합한 질문 생성
**지시사항**: 요구사항과 컨텍스트를 철저히 분석하여 스토리 품질과 팀 이해를 개선할 수 있는 모든 영역을 식별하세요. 포괄적인 user story 개발을 보장하기 위해 적극적으로 질문하세요.

**중요**: 스토리 품질에 영향을 줄 수 있는 모호성이나 누락된 세부사항이 있을 때는 기본적으로 질문하세요. 불완전하거나 불분명한 스토리를 만드는 것보다 너무 많은 질문을 하는 것이 낫습니다.

**질문 형식 규칙은 `common/question-format-guide.md` 참조**

- [Answer]: 태그 형식을 사용하여 질문 포함
- 모호성, 누락된 정보 또는 명확화가 필요한 영역에 중점
- 사용자 입력이 스토리 생성 결정을 개선할 수 있는 곳에 질문 생성
- **의심스러울 때는 질문하세요** - 과신은 나쁜 스토리로 이어집니다

**평가할 질문 카테고리** (모든 카테고리 고려):
- **사용자 페르소나** - 사용자 유형, 역할, 특성 및 동기에 대해 질문
- **스토리 세분화** - 적절한 세부 수준, 스토리 크기 및 분해 접근법에 대해 질문
- **스토리 형식** - 형식 선호도, 템플릿 사용 및 문서화 표준에 대해 질문
- **분해 접근법** - 조직 방법, 우선순위 지정 및 그룹화 전략에 대해 질문
- **수락 기준** - 세부 수준, 형식, 테스트 접근법 및 검증 방법에 대해 질문
- **사용자 여정** - 사용자 워크플로우, 상호작용 패턴 및 경험 흐름에 대해 질문
- **비즈니스 컨텍스트** - 비즈니스 목표, 성공 지표 및 이해관계자 요구에 대해 질문
- **기술적 제약** - 기술적 제한, 통합 요구사항 및 시스템 경계에 대해 질문

  - [ ] 페르소나를 관련 user stories에 매핑

## 단계 5: 스토리 옵션 제시
- 계획 문서에 스토리 분해를 위한 다양한 접근법 포함:
  - **사용자 여정 기반**: 사용자 워크플로우와 상호작용을 따르는 스토리
  - **기능 기반**: 시스템 기능과 역량을 중심으로 조직된 스토리
  - **페르소나 기반**: 다양한 사용자 유형과 그들의 요구에 따라 그룹화된 스토리
  - **도메인 기반**: 비즈니스 도메인이나 컨텍스트를 중심으로 조직된 스토리
  - **Epic 기반**: 하위 스토리가 있는 계층적 epic으로 구조화된 스토리
- 각 접근법의 장단점과 이점 설명
- 명확한 결정 기준이 있는 하이브리드 접근법 허용

## 단계 9: 답변 분석 (필수)
진행하기 전에 모든 사용자 답변을 다음에 대해 신중히 검토해야 합니다:
- **모호하거나 애매한 응답**: "혼합", "중간 어디쯤", "확실하지 않음", "상황에 따라", "아마도", "아마"
- **정의되지 않은 기준이나 용어**: 명확한 정의 없이 개념을 참조
- **모순된 답변**: 서로 충돌하는 응답
- **누락된 생성 세부사항**: 구현을 위한 구체적인 지침이 부족한 답변
- **옵션을 결합하는 답변**: 명확한 결정 규칙 없이 다른 접근법을 병합하는 응답
- **불완전한 설명**: 정의하지 않고 외부 요소를 참조하는 답변
- **가정 기반 응답**: 명시적으로 언급되지 않은 지식을 가정하는 답변

- **중요**: 철저하게 - 모든 불분명한 응답에 대해 후속 질문하세요
- 필요한 후속 질문의 예:
  - "A와 B의 '혼합'이라고 하셨는데 - A와 B를 언제 사용할지 결정하는 구체적인 기준은 무엇인가요?"
  - "A와 B 사이 '어디쯤'이라고 하셨는데 - 정확한 중간 지점 접근법을 정의해 주실 수 있나요?"
  - "'확실하지 않다'고 하셨는데 - 결정하는 데 도움이 될 추가 정보는 무엇인가요?"
  - "'복잡성에 따라'라고 하셨는데 - 복잡성 수준과 임계값을 어떻게 정의하시나요?"
  - "'하이브리드 접근법'을 선택하셨는데 - 각 방법을 언제 사용할지에 대한 구체적인 규칙은 무엇인가요?"
  - "'아마 X'라고 하셨는데 - 확실히 X vs 확실히 X가 아닌 것을 만드는 요소는 무엇인가요?"
  - "'표준 관행'을 언급하셨는데 - 그 표준 관행이 무엇인지 정의해 주실 수 있나요?"
