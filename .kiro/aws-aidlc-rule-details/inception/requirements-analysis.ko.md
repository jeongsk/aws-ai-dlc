# Requirements Analysis (적응형)

**역할 가정**: 제품 소유자

**적응형 단계**: 항상 실행됩니다. 세부 수준은 문제 복잡성에 따라 조정됩니다.

**적응형 깊이 설명은 [depth-levels.md](../common/depth-levels.md) 참조**

## 전제 조건
- Workspace Detection이 완료되어야 함
- Reverse Engineering이 완료되어야 함 (brownfield인 경우)

## 실행 단계

### 1단계: Reverse Engineering 컨텍스트 로드 (가능한 경우)

**brownfield 프로젝트인 경우**:
- `aidlc-docs/inception/reverse-engineering/architecture.md` 로드
- `aidlc-docs/inception/reverse-engineering/component-inventory.md` 로드
- `aidlc-docs/inception/reverse-engineering/technology-stack.md` 로드
- 요청 분석 시 기존 시스템을 이해하기 위해 이들을 사용

### 2단계: 사용자 요청 분석 (의도 분석)

#### 2.1 요청 명확성
- **명확함**: 구체적이고, 잘 정의되고, 실행 가능함
- **모호함**: 일반적이고, 애매하고, 명확화 필요
- **불완전함**: 주요 정보 누락

#### 2.2 요청 유형
- **새로운 기능**: 새로운 기능 추가
- **버그 수정**: 기존 문제 수정
- **리팩토링**: 코드 구조 개선
- **업그레이드**: 의존성이나 프레임워크 업데이트
- **마이그레이션**: 다른 기술로 이동
- **개선**: 기존 기능 향상
- **새 프로젝트**: 처음부터 시작

#### 2.3 초기 범위 추정
- **단일 파일**: 하나의 파일 변경
- **단일 컴포넌트**: 하나의 컴포넌트/패키지 변경
- **다중 컴포넌트**: 여러 컴포넌트에 걸친 변경
- **시스템 전체**: 전체 시스템에 영향을 미치는 변경
- **시스템 간**: 여러 시스템에 영향을 미치는 변경

#### 2.4 초기 복잡성 추정
- **사소함**: 간단하고 직관적인 변경
- **단순함**: 명확한 구현 경로
- **보통**: 일부 복잡성, 여러 고려사항
- **복잡함**: 상당한 복잡성, 많은 고려사항

### 3단계: Requirements 깊이 결정

**요청 분석을 기반으로 깊이 결정:**

**최소 깊이** - 다음의 경우 사용:
- 요청이 명확하고 단순함
- 상세한 요구사항이 필요하지 않음
- 기본 이해만 문서화

**표준 깊이** - 다음의 경우 사용:
- 요청에 명확화가 필요함
- 기능적 및 비기능적 요구사항 필요
- 일반적인 복잡성

**포괄적 깊이** - 다음의 경우 사용:
- 여러 이해관계자가 있는 복잡한 프로젝트
- 고위험 또는 중요한 시스템
- 추적성을 갖춘 상세한 요구사항 필요

### 4단계: 현재 요구사항 평가

사용자가 제공한 모든 것을 분석:
   - 의도 진술이나 설명 (이미 audit.md에 기록됨)
   - 기존 요구사항 문서 (언급된 경우 workspace 검색)
   - 붙여넣은 내용이나 파일 참조
   - 비markdown 문서를 markdown 형식으로 변환

### 5단계: 철저한 완전성 분석

**중요**: 포괄적인 분석을 사용하여 요구사항 완전성을 평가합니다. 모호성이나 누락된 세부사항이 있을 때는 기본적으로 질문하세요.

**필수**: 다음 모든 영역을 평가하고 불분명한 것이 있으면 질문하세요:
- **기능적 요구사항**: 핵심 기능, 사용자 상호작용, 시스템 동작
- **비기능적 요구사항**: 성능, 보안, 확장성, 사용성
- **사용자 시나리오**: 사용 사례, 사용자 여정, 엣지 케이스, 오류 시나리오
- **비즈니스 컨텍스트**: 목표, 제약사항, 성공 기준, 이해관계자 요구
- **기술적 컨텍스트**: 통합 지점, 데이터 요구사항, 시스템 경계
- **품질 속성**: 신뢰성, 유지보수성, 테스트 가능성, 접근성

**의심스러울 때는 질문하세요** - 불완전한 요구사항은 잘못된 구현으로 이어집니다.

### 5.1단계: Extension 적용 가능성 질문

**필수**: 로드된 모든 extension 파일에서 `## Applicability Question` 섹션을 스캔합니다. 이를 선언하는 각 extension에 대해 6단계에서 생성되는 명확화 질문 파일에 해당 질문을 포함하세요. 답변을 받은 후, `aidlc-docs/aidlc-state.md`의 `## Extension Configuration` 하에 각 extension의 활성화 상태를 기록하세요:

```markdown
## Extension Configuration
| Extension | Enabled | Decided At |
|---|---|---|
| [Extension 이름] | [Yes/No] | Requirements Analysis |
```

### 6단계: 명확화 질문 생성 (적극적 접근)
   - 요구사항이 예외적으로 명확하고 완전하지 않은 한 **항상** `aidlc-docs/inception/requirements/requirement-verification-questions.md` 생성
   - 누락되거나 불분명하거나 모호한 영역에 대해 질문
   - 기능적 요구사항, 비기능적 요구사항, 사용자 시나리오, 비즈니스 컨텍스트에 중점
   - 사용자에게 질문 문서에서 직접 모든 [Answer]: 태그를 채우도록 요청
   - 답변에 대한 다중 선택 옵션을 제시하는 경우:
     - 옵션을 A, B, C, D 등으로 라벨링
     - 옵션이 상호 배타적이고 겹치지 않도록 보장
     - **항상** 사용자 정의 응답 옵션 포함: "X) 기타 (아래 [Answer]: 태그 뒤에 설명해 주세요)"
   - 문서에서 사용자 답변 대기
   - **필수**: 모든 답변의 모호성을 분석하고 필요한 경우 후속 질문 생성
   - **필수**: 모든 모호성이 해결되거나 사용자가 명시적으로 진행을 요청할 때까지 계속 질문

### ⛔ 게이트: 사용자 답변 대기
모든 requirement-verification-questions.md의 질문이 답변되고 검증될 때까지 7단계로 진행하지 마세요.
사용자에게 질문 파일을 제시하고 중단하세요.

### 7단계: Requirements 문서 생성
   - **전제 조건**: 6단계 게이트를 통과해야 함 — 모든 답변을 받고 분석 완료
   - `aidlc-docs/inception/requirements/requirements.md` 생성
   - 상단에 의도 분석 요약 포함:
     - 사용자 요청
     - 요청 유형
     - 범위 추정
     - 복잡성 추정
   - 기능적 및 비기능적 요구사항 모두 포함
   - 명확화 질문에 대한 사용자 답변 통합
   - 주요 요구사항의 간략한 요약 제공

### 8단계: 상태 추적 업데이트

`aidlc-docs/aidlc-state.md` 업데이트:

```markdown
## 단계 진행
### 🔵 INCEPTION PHASE
- [x] Workspace Detection
- [x] Reverse Engineering (해당하는 경우)
- [x] Requirements Analysis
```

### 9단계: 기록 및 진행
   - 타임스탬프와 함께 `aidlc-docs/audit.md`에 승인 프롬프트 기록
   - 다음 구조로 완료 메시지 제시:
     1. **완료 공지** (필수): 항상 다음으로 시작:

```markdown
# 🔍 Requirements Analysis 완료
```

     2. **AI 요약** (선택사항): 요구사항의 구조화된 불릿 포인트 요약 제공
        - 형식: "Requirements analysis가 [프로젝트 유형/복잡성]을 식별했습니다:"
        - 주요 기능적 요구사항 나열 (불릿 포인트)
        - 주요 비기능적 요구사항 나열 (불릿 포인트)
        - 관련된 경우 아키텍처 고려사항이나 기술적 결정 언급
        - 워크플로우 지침 포함하지 마세요 ("검토해 주세요", "알려주세요", "다음 단계로 진행", "진행하기 전에")
        - 사실적이고 내용 중심으로 유지
     3. **형식화된 워크플로우 메시지** (필수): 항상 다음 정확한 형식으로 끝내기:

```markdown
> **📋 <u>**검토 필요:**</u>**  
> requirements 문서를 검토해 주세요: `aidlc-docs/inception/requirements/requirements.md`



> **🚀 <u>**다음 단계:**</u>**
>
> **다음을 선택할 수 있습니다:**
>
> 🔧 **변경 요청** - 검토를 바탕으로 필요한 경우 requirements 수정 요청
> [User Stories가 건너뛰어질 경우, 이 옵션 추가:]
> 📝 **User Stories 추가** - **User Stories** 단계 포함 선택 (프로젝트 단순성으로 인해 현재 건너뛰어짐)  
> ✅ **승인 및 계속** - requirements를 승인하고 **[User Stories/Workflow Planning]**로 진행

---
```

**참고**: User Stories 단계가 건너뛰어질 때만 "Add User Stories" 옵션을 포함하세요. [User Stories/Workflow Planning]을 실제 다음 단계 이름으로 교체하세요.

   - 명시적 사용자 승인을 기다린 후 진행
   - 타임스탬프와 함께 승인 응답 기록
   - aidlc-state.md에서 Requirements Analysis 단계를 완료로 업데이트
