# Units Generation - 상세 단계

## 개요
이 단계는 두 개의 통합된 부분을 통해 시스템을 관리 가능한 작업 단위로 분해합니다:
- **파트 1 - 계획**: 질문이 포함된 분해 계획 생성, 답변 수집, 모호성 분석, 승인 획득
- **파트 2 - 생성**: 승인된 계획을 실행하여 단위 산출물 생성

**정의**: 작업 단위는 개발 목적을 위한 스토리의 논리적 그룹입니다. 마이크로서비스의 경우 각 단위는 독립적으로 배포 가능한 서비스가 됩니다. 모놀리스의 경우 단일 단위는 논리적 모듈이 있는 전체 애플리케이션을 나타냅니다.

**용어**: 독립적으로 배포 가능한 컴포넌트에는 "Service", 서비스 내 논리적 그룹에는 "Module", 계획 컨텍스트에는 "Unit of Work"를 사용합니다.

## 전제 조건
- Context Assessment가 완료되어야 함
- Requirements Assessment 권장 (기능적 범위 제공)
- Story Development 권장 (스토리가 단위에 매핑됨)
- Application Design 단계 필수 (컴포넌트, 메서드, 서비스 결정)
- 실행 계획에서 Design 단계를 실행해야 한다고 표시되어야 함

# 파트 1: 계획

## 단계 1: 작업 단위 계획 생성
- 시스템을 작업 단위로 분해하기 위한 체크박스 []가 있는 계획 생성
- 시스템을 관리 가능한 개발 단위로 분해하는 데 중점
- 각 단계와 하위 단계에는 체크박스 []가 있어야 함

## 단계 2: 계획에 필수 단위 산출물 포함
**항상** 단위 계획에 다음 필수 산출물을 포함:
- [ ] 단위 정의와 책임이 포함된 `aidlc-docs/inception/application-design/unit-of-work.md` 생성
- [ ] 의존성 매트릭스가 포함된 `aidlc-docs/inception/application-design/unit-of-work-dependency.md` 생성
- [ ] 스토리를 단위에 매핑하는 `aidlc-docs/inception/application-design/unit-of-work-story-map.md` 생성
- [ ] **Greenfield만**: `unit-of-work.md`에 코드 조직 전략 문서화 (구조 패턴은 code-generation.md 참조)
- [ ] 단위 경계와 의존성 검증
- [ ] 모든 스토리가 단위에 할당되었는지 확인

## 단계 3: 컨텍스트에 적합한 질문 생성
**지시사항**: 요구사항, 스토리, application design을 분석하여 이 특정 분해 문제와 관련된 질문만 생성하세요. 아래 카테고리를 영감으로 사용하되, 필수 체크리스트로 사용하지 마세요. 해당되지 않는 전체 카테고리는 건너뛰세요.

- [Answer]: 태그 형식을 사용하여 질문 포함
- 이 컨텍스트에 특정한 모호성과 누락된 정보에 중점
- 의사결정에 사용자 입력이 필요한 경우에만 질문 생성

**예시 질문 카테고리** (필요에 따라 조정):
- **스토리 그룹화** - 여러 스토리가 존재하고 그룹화 전략이 불분명한 경우에만
- **의존성** - 여러 단위가 있을 가능성이 있고 통합 접근 방식이 모호한 경우에만
- **팀 정렬** - 팀 구조나 소유권이 불분명한 경우에만
- **기술적 고려사항** - 단위 간 확장성/배포 요구사항이 다른 경우에만
- **비즈니스 도메인** - 도메인 경계나 bounded context가 불분명한 경우에만
- **코드 조직 (Greenfield 다중 단위만)** - 배포 모델과 디렉토리 구조 선호도에 대해 질문

## 단계 4: UOW 계획 저장
- `aidlc-docs/inception/plans/unit-of-work-plan.md`로 저장
- 사용자 입력을 위한 모든 [Answer]: 태그 포함
- 계획이 시스템 분해의 모든 측면을 다루도록 보장

## 단계 5: 사용자 입력 요청
- 사용자에게 계획 문서에서 직접 [Answer]: 태그를 채우도록 요청
- 분해 결정의 중요성 강조
- [Answer]: 태그 완성에 대한 명확한 지침 제공

## 단계 6: 답변 수집
- 사용자가 문서의 [Answer]: 태그를 사용하여 모든 질문에 답변할 때까지 대기
- 모든 [Answer]: 태그가 완성될 때까지 진행하지 마세요
- 문서를 검토하여 [Answer]: 태그가 비어있지 않은지 확인

## 단계 7: 답변 분석 (필수)
진행하기 전에 모든 사용자 답변을 다음 사항에 대해 신중히 검토해야 합니다:
- **모호하거나 애매한 응답**: "혼합", "중간 정도", "확실하지 않음", "상황에 따라"
- **정의되지 않은 기준이나 용어**: 명확한 정의 없이 개념을 참조
- **모순된 답변**: 서로 충돌하는 응답
- **누락된 생성 세부사항**: 구체적인 지침이 부족한 답변
- **옵션을 결합한 답변**: 명확한 결정 규칙 없이 다른 접근 방식을 병합한 응답

## 단계 8: 필수 후속 질문
7단계의 분석에서 모호한 답변이 발견되면 반드시:
- [Answer]: 태그를 사용하여 계획 문서에 구체적인 후속 질문 추가
- 모든 모호성이 해결될 때까지 승인으로 진행하지 마세요
- 필요한 후속 질문의 예:
  - "'A와 B의 혼합'이라고 하셨는데 - A와 B 중 언제 사용할지 결정하는 구체적인 기준은 무엇인가요?"
  - "'A와 B 사이 어딘가'라고 하셨는데 - 정확한 중간 지점 접근 방식을 정의할 수 있나요?"
  - "'확실하지 않다'고 하셨는데 - 결정하는 데 도움이 될 추가 정보는 무엇인가요?"
  - "'복잡성에 따라'라고 하셨는데 - 복잡성 수준을 어떻게 정의하나요?"

## 단계 9: 승인 요청
- 질문: "**작업 단위 계획이 완료되었습니다. aidlc-docs/inception/plans/unit-of-work-plan.md에서 계획을 검토하세요. 생성으로 진행할 준비가 되었나요?**"
- 사용자가 확인할 때까지 진행하지 마세요

- 완전한 승인 프롬프트 텍스트 포함

## 중요한 규칙

### 계획 단계 규칙
- 컨텍스트와 관련된 질문만 생성
- 모든 질문에 [Answer]: 태그 형식 사용
- 진행하기 전에 모든 답변을 모호성에 대해 분석
- 후속 질문으로 모든 모호성 해결
- 생성하기 전에 명시적인 사용자 승인 획득

### 생성 단계 규칙
- **하드코딩된 로직 없음**: 작업 단위 계획에 작성된 것만 실행
- **계획을 정확히 따르기**: 단계 순서에서 벗어나지 마세요
- **체크박스 업데이트**: 각 단계 완료 후 즉시 [x]로 표시
- **승인된 접근 방식 사용**: 계획의 분해 방법론 따르기
- **완료 확인**: 진행하기 전에 모든 단위 산출물이 완료되었는지 확인

- 계획에 따라 모든 단위 산출물이 생성됨:
