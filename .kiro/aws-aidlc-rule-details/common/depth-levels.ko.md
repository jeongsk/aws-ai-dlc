# 적응형 깊이

**목적**: AI-DLC가 문제 복잡성에 따라 세부 수준을 어떻게 적응시키는지 설명

## 핵심 원칙

**단계가 실행될 때, 정의된 모든 산출물이 생성됩니다. "깊이"는 해당 산출물 내의 세부 수준과 엄격성을 의미하며, 이는 문제의 복잡성에 적응합니다.**

## 단계 선택 vs 세부 수준

### 단계 선택 (이진)
- **Workflow Planning**이 결정: 각 단계에 대해 실행 또는 건너뛰기
- **실행하는 경우**: 단계가 실행되고 정의된 모든 산출물을 생성
- **건너뛰는 경우**: 단계가 전혀 실행되지 않음

### 세부 수준 (적응형)
- **간단한 문제**: 필수 세부사항이 포함된 간결한 산출물
- **복잡한 문제**: 광범위한 세부사항이 포함된 포괄적 산출물
- **모델이 결정**: 규범적 규칙이 아닌 문제 특성에 기반

## 세부 수준에 영향을 미치는 요소

모델은 적절한 세부 수준을 결정할 때 다음 요소들을 고려합니다:

1. **요청 명확성**: 사용자의 요청이 얼마나 명확하고 완전한가?
2. **문제 복잡성**: 솔루션 공간이 얼마나 복잡한가?
3. **범위**: 단일 파일, 컴포넌트, 다중 컴포넌트, 또는 시스템 전체?
4. **위험 수준**: 오류나 누락의 영향은 무엇인가?
5. **사용 가능한 컨텍스트**: Greenfield vs brownfield, 기존 문서화
6. **사용자 선호도**: 사용자가 간결함 또는 세부사항에 대한 선호를 표현했는가?

## 예시: Requirements Analysis 산출물

**모든 시나리오에서 동일한 산출물을 생성**:
- `requirement-verification-questions.md` (필요시)
- `requirements.md`

**참고**: 사용자의 초기 요청은 `audit.md`에 캡처됨 (별도의 user-intent.md 불필요)

**세부 수준은 복잡성에 따라 달라짐**:

### 간단한 시나리오 (버그 수정)
- **requirement-verification-questions.md**: 필요한 명확화 질문
- **requirements.md**: 간결한 기능 요구사항, 최소 섹션

### 복잡한 시나리오 (시스템 마이그레이션)
- **requirement-verification-questions.md**: 여러 라운드, 10개 이상의 질문
- **requirements.md**: 포괄적인 기능 + 비기능 요구사항, 추적성, 수용 기준

## 예시: Application Design 산출물

**모든 시나리오에서 동일한 산출물을 생성**:
- `application-design.md`
- `component-diagram.md`

**세부 수준은 복잡성에 따라 달라짐**:

### 간단한 시나리오 (단일 컴포넌트)
- **application-design.md**: 기본 컴포넌트 설명, 주요 메서드
- **component-diagram.md**: 필수 관계가 포함된 간단한 다이어그램

### 복잡한 시나리오 (다중 컴포넌트 시스템)
- **application-design.md**: 상세한 컴포넌트 책임, 시그니처가 포함된 모든 메서드, 디자인 패턴, 고려된 대안
- **component-diagram.md**: 모든 관계, 데이터 흐름, 통합 지점이 포함된 포괄적 다이어그램

## 모델을 위한 지침 원칙

**"당면한 문제에 필요한 정확한 세부사항을 생성하라 - 그 이상도 이하도 아니다."**

- 간단한 문제를 불필요한 세부사항으로 인위적으로 부풀리지 마라
- 중요한 세부사항을 생략하여 복잡한 문제를 부당하게 축소하지 마라
- 문제 특성이 세부 수준을 자연스럽게 주도하도록 하라
- 단계가 실행될 때 필요한 모든 산출물은 항상 생성된다
