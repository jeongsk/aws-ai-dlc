# 과신 방지 가이드

## 문제 설명

AI-DLC는 복잡한 프로젝트 의도 설명에 대해서도 충분한 명확화 질문을 하지 않아 과신을 보였습니다. 이로 인해 적절한 요구사항을 수집하는 대신 가정을 하게 되었습니다.

## 근본 원인 분석

과신 문제는 질문을 건너뛰도록 권장하는 여러 단계의 지시사항으로 인해 발생했습니다:

1. **Functional Design**: "해당되지 않으면 전체 카테고리 건너뛰기"
2. **User Stories**: "카테고리를 영감으로 사용, 필수 체크리스트가 아님"
3. **Requirements Analysis**: 최소한의 질문을 권장하는 유사한 패턴
4. **NFR Requirements**: 철저한 분석을 저해하는 "~인 경우에만" 조건

이러한 지시사항들은 AI가 포괄적인 요구사항 수집을 권장하기보다는 질문을 피하도록 지시했습니다.

## 구현된 해결책

### 업데이트된 질문 생성 철학

**기존 접근법**: "절대적으로 필요한 경우에만 질문하기"
**새로운 접근법**: "의심스러우면 질문하기 - 과신은 좋지 않은 결과를 낳습니다"

### 주요 변경사항

#### 1. Requirements Analysis 단계
- "필요한 경우에만"에서 "예외적으로 명확하지 않은 한 항상 질문 생성"으로 변경
- 포괄적 평가 영역 추가 (기능적, 비기능적, 비즈니스 컨텍스트, 기술적 컨텍스트)
- 사전 예방적 질문 접근법 강조

#### 2. User Stories 단계
- "전체 카테고리 건너뛰기" 지시사항 제거
- 평가할 포괄적 질문 카테고리 추가
- 답변 분석 요구사항 강화
- 후속 질문 의무 강화

#### 3. Functional Design 단계
- "~인 경우에만" 조건을 포괄적 평가로 대체
- 더 많은 질문 카테고리 추가 (데이터 플로우, 통합 지점, 오류 처리)
- 모호성 감지 및 해결 요구사항 강화

#### 4. NFR Requirements 단계
- 기본 NFR을 넘어선 질문 카테고리 확장
- 신뢰성, 유지보수성, 사용성 고려사항 추가
- 기술적 모호성에 대한 답변 분석 강화

### 새로운 지침 원칙

1. **질문하기 기본**: 모호성이 있을 때는 명확화 질문하기
2. **포괄적 커버리지**: 모든 관련 카테고리 평가, 영역 건너뛰지 않기
3. **철저한 분석**: 모호성에 대한 모든 사용자 응답을 신중히 분석
4. **필수 후속 조치**: 불분명한 응답에 대해 후속 질문 생성
5. **모호성과 함께 진행 금지**: 모든 모호성이 해결될 때까지 앞으로 진행하지 않기

## 구현 가이드라인

### 질문 생성을 위해
- 모든 질문 카테고리 평가, 어떤 것도 건너뛰지 않기
- 명확화가 품질을 향상시킬 수 있는 곳에서 질문하기
- 각 단계에 포괄적 질문 카테고리 포함
- 질문 제외보다는 포함을 기본으로 하기

### 답변 분석을 위해
- 모호한 응답 찾기: "상황에 따라", "아마도", "확실하지 않음", "혼합", "그 사이 어딘가"
- 정의되지 않은 용어와 외부 개념에 대한 참조 감지
- 모순되거나 불완전한 답변 식별
- 모든 모호성에 대해 후속 질문 생성

### 후속 질문을 위해
- 모호성이 감지되면 별도의 명확화 파일 생성
- 각 모호성을 해결하기 위한 구체적 질문하기
- 모든 불분명한 응답이 명확해질 때까지 진행하지 않기
- 철저하게 - 명확화 부족보다는 과도한 명확화가 낫습니다

## 품질 보증

### 주의해야 할 위험 신호
- 복잡한 프로젝트에서 질문 없이 단계 완료
- 모호하거나 애매한 사용자 응답으로 진행
- 정당화 없이 전체 질문 카테고리 건너뛰기
- 명확화를 요청하는 대신 가정하기

### 성공 지표
- 프로젝트 복잡성에 적절한 수의 명확화 질문
- 필요시 후속 조치와 함께 사용자 응답의 철저한 분석
- 구현으로 진행하기 전에 명확하고 모호하지 않은 요구사항
- 사전 명확화 개선으로 인한 후속 단계에서의 변경 필요성 감소

## 유지보수

이 가이드는 다음과 같은 경우에 참조해야 합니다:
- AI-DLC에 새로운 단계 추가
- 기존 단계 지시사항 업데이트
- 과신 문제에 대한 AI-DLC 성능 검토
- AI-DLC 질문 생성 원칙에 대한 팀 구성원 교육

## 핵심 요점

**잘못된 가정을 하는 것보다는 너무 많은 질문을 하는 것이 낫습니다.** 사전에 명확화 질문을 하는 비용은 가정에 기반한 잘못된 솔루션을 구현하는 비용보다 훨씬 적습니다.
