# 질문 형식 가이드

## 필수: 모든 질문은 이 형식을 사용해야 함

### 규칙: 채팅에서 직접 질문하지 마세요
**중요**: 채팅에서 직접 질문하면 절대 안 됩니다. 모든 질문은 전용 질문 파일에 배치되어야 합니다.

### 질문 파일 형식

#### 파일 명명 규칙
- 설명적인 이름 사용: `{phase-name}-questions.md`
- 예시:
  - `classification-questions.md`
  - `requirements-questions.md`
  - `story-planning-questions.md`
  - `design-questions.md`

#### 질문 구조
모든 질문은 의미 있는 옵션과 마지막 옵션으로 "기타"를 포함해야 합니다:

```markdown
## 질문 [번호]
[명확하고 구체적인 질문 텍스트]

A) [첫 번째 의미 있는 옵션]
B) [두 번째 의미 있는 옵션]
[...필요에 따라 추가 옵션...]
X) 기타 (아래 [Answer]: 태그 뒤에 설명해 주세요)

[Answer]: 
```

**중요**: 
- "기타"는 모든 질문의 마지막 옵션으로 필수입니다
- 의미 있는 옵션만 포함하세요 - 슬롯을 채우기 위해 옵션을 만들어내지 마세요
- 의미가 있는 만큼 많거나 적은 옵션을 사용하세요 (최소 2개 + 기타)

### 사용자 응답 형식
사용자는 [Answer]: 태그 뒤에 문자 선택을 채워서 답합니다:

```markdown
## 질문 1
주요 사용자 인증 방법은 무엇입니까?

A) 사용자명과 비밀번호
B) 소셜 미디어 로그인 (Google, Facebook)
C) Single Sign-On (SSO)
D) Multi-factor authentication

[Answer]: C
```

### 사용자 응답 읽기
사용자가 완료를 확인한 후:
1. 질문 파일 읽기
2. [Answer]: 태그 뒤의 답변 추출
3. 모든 질문이 답변되었는지 검증
4. 응답을 기반으로 분석 진행

### 객관식 가이드라인

#### 옵션 수
- 최소: 2개의 의미 있는 옵션 + "기타" (A, B, C)
- 일반적: 3-4개의 의미 있는 옵션 + "기타" (A, B, C, D, E)
- 최대: 5개의 의미 있는 옵션 + "기타" (A, B, C, D, E, F)
- **중요**: 슬롯을 채우기 위해 옵션을 만들어내지 마세요 - 의미 있는 선택만 포함하세요

#### 옵션 품질
- 옵션을 상호 배타적으로 만드세요
- 가장 일반적인 시나리오를 다루세요
- 의미 있고 현실적인 옵션만 포함하세요
- **항상 "기타"를 마지막 옵션으로 포함하세요** (필수)
- 구체적이고 명확하게 하세요
- **A, B, C, D 슬롯을 채우기 위해 옵션을 만들어내지 마세요**

### 워크플로우 통합

#### 1단계: 질문 파일 생성
```markdown
모든 질문이 포함된 aidlc-docs/{phase-name}-questions.md 생성
```

#### 2단계: 사용자에게 알림
```
"[X]개의 질문이 포함된 {phase-name}-questions.md를 생성했습니다. 
[Answer]: 태그 뒤에 문자 선택을 채워서 각 질문에 답해 주세요. 
옵션 중 어느 것도 귀하의 요구에 맞지 않으면 마지막 옵션(기타)을 선택하고 선호사항을 설명해 주세요. 완료되면 알려주세요."
```

#### 3단계: 확인 대기
사용자가 "완료", "끝", "마침" 또는 유사한 말을 할 때까지 기다립니다.

#### 4단계: 읽기 및 분석
```
aidlc-docs/{phase-name}-questions.md 읽기
모든 답변 추출
완전성 검증
분석 진행
```

### 오류 처리

#### 누락된 답변
[Answer]: 태그가 비어있는 경우:
```
"질문 [X]가 답변되지 않았습니다. 진행하기 전에 
모든 질문에 대해 문자 선택 중 하나를 사용하여 답변을 제공해 주세요."
```

#### 잘못된 답변
답변이 유효한 문자 선택이 아닌 경우:
```
"질문 [X]에 잘못된 답변 '[answer]'이 있습니다. 
질문에 제공된 문자 선택만 사용해 주세요."
```

#### 모호한 답변
사용자가 문자 대신 설명을 제공한 경우:
```
"질문 [X]에 대해 귀하의 답변과 가장 잘 맞는 문자 선택을 제공해 주세요. 
어느 것도 맞지 않으면 '기타'를 선택하고 [Answer]: 태그 뒤에 설명을 추가해 주세요."
```

### 모순 및 모호성 감지

**필수**: 사용자 응답을 읽은 후, 모순과 모호성을 확인해야 합니다.

#### 모순 감지
논리적으로 일관되지 않는 답변 찾기:
- 범위 불일치: "버그 수정"이지만 "전체 codebase 영향"
- 위험 불일치: "낮은 위험"이지만 "Breaking changes"
- 일정 불일치: "빠른 수정"이지만 "여러 subsystems"
- 영향 불일치: "단일 component"이지만 "상당한 아키텍처 변경"

#### 모호성 감지
불명확하거나 경계선상의 응답 찾기:
- 여러 분류에 맞을 수 있는 답변
- 구체성이 부족한 응답
- 여러 질문에 걸쳐 상충하는 지표

#### 명확화 질문 생성
모순이나 모호성이 감지된 경우:

1. **명확화 파일 생성**: `{phase-name}-clarification-questions.md`
2. **문제 설명**: 어떤 모순/모호성이 감지되었는지 명확히 설명
3. **타겟 질문하기**: 문제를 해결하기 위해 객관식 형식 사용
4. **원래 질문 참조**: 어떤 질문에 상충하는 답변이 있었는지 보여주기

**예시**:
```markdown
# [Phase Name] 명확화 질문

귀하의 응답에서 명확화가 필요한 모순을 감지했습니다:

## 모순 1: [간단한 설명]
"[답변 A]" (Q[X]:[문자])라고 표시했지만 "[답변 B]" (Q[Y]:[문자])라고도 했습니다.
이러한 응답은 [설명] 때문에 모순됩니다.

### 명확화 질문 1
[모순을 해결하기 위한 구체적인 질문]

A) [첫 번째 답변 쪽으로 해결하는 옵션]
B) [두 번째 답변 쪽으로 해결하는 옵션]
C) [중간 지점을 제공하는 옵션]
D) [질문을 재구성하는 옵션]

[Answer]: 

## 모호성 1: [간단한 설명]
Q[X]에 대한 귀하의 응답 ("[답변]")은 [설명] 때문에 모호합니다.

### 명확화 질문 2
[모호성을 명확히 하기 위한 구체적인 질문]

A) [명확한 옵션 1]
B) [명확한 옵션 2]
C) [명확한 옵션 3]
D) [명확한 옵션 4]

[Answer]: 
```

#### 명확화를 위한 워크플로우

1. **감지**: 모든 응답을 모순/모호성에 대해 분석
2. **생성**: 문제가 발견되면 명확화 질문 파일 생성
3. **알림**: 사용자에게 문제와 명확화 파일에 대해 알림
4. **대기**: 사용자가 명확화를 제공할 때까지 진행하지 않음
5. **재검증**: 명확화 후 일관성을 다시 확인
6. **진행**: 모든 모순이 해결된 후에만 앞으로 진행

#### 사용자 메시지 예시
```
"귀하의 응답에서 2개의 모순을 감지했습니다:

1. 버그 수정 범위 vs. codebase 영향 (Q1 vs Q2)
2. 낮은 위험 vs. breaking changes (Q7 vs Q4)

이를 해결하기 위해 2개의 질문이 포함된 classification-clarification-questions.md를 생성했습니다.
분류를 진행하기 전에 이러한 명확화 질문에 답해 주세요."
```

### 모범 사례

1. **구체적이어야 함**: 질문은 명확하고 모호하지 않아야 합니다
2. **포괄적이어야 함**: 모든 필요한 정보를 다뤄야 합니다
3. **간결해야 함**: 질문을 한 주제에 집중시켜야 합니다
4. **실용적이어야 함**: 옵션은 현실적이고 실행 가능해야 합니다
5. **일관성이 있어야 함**: 모든 질문 파일에서 동일한 형식을 사용해야 합니다
