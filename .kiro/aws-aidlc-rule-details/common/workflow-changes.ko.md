# 워크플로우 중간 변경 및 단계 관리

## 개요

사용자는 워크플로우 중에 실행 계획이나 단계 실행에 대한 변경을 요청할 수 있습니다. 이 문서는 이러한 요청을 안전하고 효과적으로 처리하는 방법에 대한 지침을 제공합니다.

---

## 워크플로우 중간 변경 유형

### 1. 건너뛴 단계 추가

**시나리오**: 사용자가 원래 건너뛴 단계를 추가하고 싶어함

**예시**: "실제로는 그 단계를 건너뛰었지만 user stories를 추가하고 싶습니다"

**처리 방법**:
1. **요청 확인**: "User Stories 단계를 추가하고 싶으시군요. 이것은 user stories와 personas를 생성합니다. 확인하시겠습니까?"
2. **의존성 확인**: 모든 전제 조건 단계가 완료되었는지 확인
3. **실행 계획 업데이트**: 근거와 함께 `execution-plan.md`에 단계 추가
4. **상태 업데이트**: `aidlc-state.md`에서 단계를 "PENDING"으로 표시
5. **단계 실행**: 일반적인 단계 실행 프로세스 따르기
6. **변경 기록**: 타임스탬프와 이유와 함께 `audit.md`에 문서화

**고려사항**:
- 새로운 산출물의 이익을 얻을 수 있는 후속 단계를 업데이트해야 할 수 있음
- 기존 산출물이 새로운 정보를 통합하기 위해 수정이 필요할 수 있음
- 일정이 연장됨

---

### 2. 계획된 단계 건너뛰기

**시나리오**: 사용자가 실행하도록 계획된 단계를 건너뛰고 싶어함

**예시**: "지금은 NFR Design 단계를 건너뛰겠습니다"

**처리 방법**:
1. **요청 확인**: "NFR Design을 건너뛰고 싶으시군요. 이는 NFR 패턴이나 논리적 컴포넌트가 통합되지 않음을 의미합니다. 확인하시겠습니까?"
2. **영향에 대해 경고**: 무엇이 누락될지와 잠재적 결과를 설명
3. **명시적 확인 받기**: 사용자가 영향을 이해했음을 명시적으로 확인해야 함
4. **실행 계획 업데이트**: 이유와 함께 단계를 "SKIPPED"로 표시
5. **상태 업데이트**: `aidlc-state.md`에서 단계를 "SKIPPED"로 표시
6. **후속 단계 조정**: 후속 단계가 수동 설정이 필요할 수 있음을 기록
7. **변경 기록**: 타임스탬프와 이유와 함께 `audit.md`에 문서화

**고려사항**:
- 후속 단계가 실패하거나 수동 개입이 필요할 수 있음
- 사용자가 누락된 산출물에 대한 책임을 받아들임
- 필요시 나중에 다시 추가할 수 있음

---

### 3. 현재 단계 재시작

**시나리오**: 사용자가 현재 단계 결과에 만족하지 않고 다시 하고 싶어함

**예시**: "이 user stories가 마음에 들지 않습니다. 다시 시작할 수 있을까요?"

**처리 방법**:
1. **우려사항 이해**: "stories에서 구체적으로 무엇을 변경하고 싶으신가요?"
2. **옵션 제공**:
   - **옵션 A**: 기존 산출물 수정 (더 빠름, 일부 작업 보존)
   - **옵션 B**: 완전 재시작 (깨끗한 시작, 더 많은 시간)
3. **재시작 선택 시**:
   - 기존 산출물 보관: `{artifact}.backup.{timestamp}`
   - 계획 파일의 단계 체크박스 재설정
   - `aidlc-state.md`에서 단계를 "IN PROGRESS"로 표시
   - 단계 완료 상태 지우기
   - 처음부터 재실행
4. **변경 기록**: 재시작 이유와 변경될 내용 문서화

**고려사항**:
- 기존 작업이 손실됨 (하지만 백업됨)
- 의존적인 단계를 다시 해야 할 수 있음
- 일정이 연장됨

---

### 4. 이전 단계 재시작

**시나리오**: 사용자가 완료된 단계로 돌아가서 다시 하고 싶어함

**예시**: "이전에 내린 아키텍처 결정을 변경하고 싶습니다"

**처리 방법**:
1. **영향 평가**: 재시작할 단계에 의존하는 모든 단계 식별
2. **사용자에게 경고**: "Application Design을 재시작하면 다음을 다시 해야 합니다: Units Planning, Units Generation, per-unit design (모든 units), Code Planning, Code Generation. 확인하시겠습니까?"
3. **명시적 확인 받기**: 사용자가 전체 영향을 이해해야 함
4. **확인 시**:
   - 영향받는 모든 산출물 보관
   - `aidlc-state.md`에서 영향받는 모든 단계 재설정
   - 영향받는 모든 계획 파일의 체크박스 지우기
   - 재시작할 단계로 돌아가기
   - 그 지점부터 다시 실행
5. **변경 기록**: 전체 영향과 재시작 이유 문서화

**고려사항**:
- 상당한 재작업 필요
- 모든 의존적인 단계를 다시 해야 함
- 일정이 크게 연장됨
- 재시작보다 수정이 나은지 고려

---

### 5. 단계 깊이 변경

**시나리오**: 사용자가 현재 또는 향후 단계의 깊이 수준을 변경하고 싶어함

**예시**: "표준 대신 포괄적인 requirements analysis를 해봅시다"

**처리 방법**:
1. **요청 확인**: "Requirements Analysis를 Standard에서 Comprehensive 깊이로 변경하고 싶으시군요. 더 철저하지만 시간이 더 걸립니다. 확인하시겠습니까?"
2. **실행 계획 업데이트**: `workflow-planning.md`에서 깊이 수준 변경
3. **접근 방식 조정**: 단계에 대한 comprehensive 깊이 가이드라인 따르기
4. **추정치 업데이트**: 새로운 일정 추정치를 사용자에게 알림
5. **변경 기록**: 깊이 변경과 이유 문서화

**고려사항**:
- 더 깊은 깊이 = 더 많은 시간이지만 더 나은 품질
- 더 얕은 깊이 = 더 빠르지만 세부사항을 놓칠 수 있음
- 단계 전이나 진행 중에만 변경 가능, 완료 후에는 불가

---

### 6. 워크플로우 일시정지

**시나리오**: 사용자가 일시정지하고 나중에 재개해야 함

**예시**: "지금 중단하고 내일 계속해야 합니다"

**처리 방법**:
1. **현재 단계 완료**: 가능하면 진행 중인 현재 단계 완료
2. **체크박스 업데이트**: 완료된 모든 단계를 [x]로 표시
3. **상태 업데이트**: `aidlc-state.md`가 현재 상태를 반영하도록 보장
4. **일시정지 기록**: `audit.md`에 일시정지 지점 문서화
5. **재개 지침 제공**: "돌아오시면, 기존 프로젝트를 감지하고 다음에서 계속할 것을 제안합니다: [현재 단계, 현재 단계]"

**재개 시**:
1. **기존 프로젝트 감지**: `aidlc-state.md` 확인
2. **컨텍스트 로드**: 완료된 단계의 모든 산출물 읽기
3. **상태 표시**: 현재 단계와 다음 단계 표시
4. **옵션 제공**: 중단한 곳에서 계속하거나 이전 작업 검토
5. **재개 기록**: `audit.md`에 재개 지점 문서화

---

### 7. 아키텍처 결정 변경

**시나리오**: 사용자가 monolith에서 microservices로 (또는 그 반대로) 변경하고 싶어함

**예시**: "실제로는 monolith 대신 microservices를 해봅시다"

**처리 방법**:
1. **현재 진행 상황 평가**: 워크플로우에서 얼마나 진행되었는지 결정
2. **영향 설명**: 
   - Units Planning 전: 최소 영향, 결정만 업데이트
   - Units Planning 후: Units Planning, Units Generation, 모든 per-unit design을 다시 해야 함
   - Code Generation 후: 상당한 재작업 필요
3. **접근 방식 권장**:
   - 워크플로우 초기: Application Design 단계부터 재시작
   - 워크플로우 후기: 수정이 가능한지 vs. 재시작 고려
4. **확인 받기**: 사용자가 변경의 전체 범위를 이해해야 함
5. **변경 실행**: 영향받는 단계에 대한 재시작 절차 따르기

**고려사항**:
- 아키텍처 변경은 연쇄 효과가 있음
- 워크플로우 초기 = 변경하기 쉬움
- 워크플로우 후기 = 비용 vs. 이익 고려

---

### 8. Units 추가/제거

**시나리오**: 사용자가 Units Generation 후에 units를 추가하거나 제거하고 싶어함

**예시**: "Payment unit을 Payment와 Billing으로 분할해야 합니다"

**처리 방법**:
1. **영향 평가**: 어떤 units가 design/code를 완료했는지 결정
2. **결과 설명**:
   - Unit 추가: 새 unit에 대한 전체 design과 code 필요
   - Unit 제거: 다른 units로 기능 재분배 필요
   - Unit 분할: 결과로 나오는 두 units 모두에 대한 design과 code 재작업 필요
3. **Unit 산출물 업데이트**:
   - `unit-of-work.md` 수정
   - `unit-of-work-dependency.md` 업데이트
   - `unit-of-work-story-map.md` 수정
4. **영향받는 Units 재설정**: 영향받는 units를 재설계 필요로 표시
5. **변경 실행**: 영향받는 units에 대한 일반적인 unit design과 code 프로세스 따르기

**고려사항**:
- 해당 units의 모든 downstream 단계에 영향
- 의존성이 변경되면 다른 units에 영향을 줄 수 있음
- 일정 영향은 영향받는 units 수에 따라 달라짐

## 변경 처리를 위한 일반 가이드라인

### 변경하기 전

1. **요청 이해**: 사용자가 무엇을 변경하고 싶어하는지와 이유에 대해 명확한 질문하기
2. **영향 평가**: 영향받는 모든 단계, 산출물, 의존성 식별
3. **결과 설명**: 무엇을 다시 해야 하는지와 일정 영향을 명확히 전달
4. **대안 제공**: 때로는 재시작보다 수정이 나음
5. **명시적 확인 받기**: 사용자가 영향을 이해하고 받아들여야 함

### 변경 중

1. **기존 작업 보관**: 파괴적인 변경을 하기 전에 항상 백업
2. **모든 추적 업데이트**: `aidlc-state.md`, 계획 파일, `audit.md`를 동기화 상태로 유지
3. **진행 상황 전달**: 무엇이 일어나고 있는지 사용자에게 계속 알림
4. **변경 검증**: 모든 산출물에서 변경이 일관되도록 보장
5. **연속성 테스트**: 변경 후 워크플로우가 원활하게 계속될 수 있는지 확인

### 변경 후

1. **일관성 확인**: 모든 산출물이 변경과 일치하는지 확인
2. **문서 업데이트**: 모든 참조가 업데이트되었는지 보장
3. **완전히 기록**: `audit.md`에 전체 변경 이력 문서화
4. **사용자와 확인**: 변경이 사용자의 기대를 충족하는지 확인
5. **워크플로우 재개**: 새로운 상태에서 일반적인 실행으로 계속

## 변경 요청 결정 트리

```
사용자가 변경 요청
    |
    ├─ 현재 단계인가?
    |   ├─ 예: 현재 단계를 수정하거나 재시작할 수 있음
    |   └─ 아니오: 다음 질문으로
    |
    ├─ 완료된 단계인가?
    |   ├─ 예: 의존적인 단계에 대한 영향 평가
    |   |   ├─ 낮은 영향: 수정하고 의존적인 것들 업데이트
    |   |   └─ 높은 영향: 해당 단계부터 재시작 권장
    |   └─ 아니오: 다음 질문으로
    |
    ├─ 건너뛴 단계를 추가하는 것인가?
    |   ├─ 예: 전제 조건 확인, 계획에 추가, 실행
    |   └─ 아니오: 다음 질문으로
    |
    ├─ 계획된 단계를 건너뛰는 것인가?
    |   ├─ 예: 영향에 대해 경고, 확인 받기, 건너뛰기
    |   └─ 아니오: 다음 질문으로
    |
    └─ 깊이 수준을 변경하는 것인가?
        ├─ 예: 계획 업데이트, 접근 방식 조정
        └─ 아니오: 사용자와 요청 명확화
```

## 모범 사례

1. **항상 확인**: 명시적인 사용자 확인 없이 파괴적인 변경을 하지 마세요
2. **영향 설명**: 사용자가 결정하기 전에 결과를 이해해야 합니다
3. **옵션 제공**: 때로는 변경을 처리하는 여러 방법이 있습니다
4. **먼저 보관**: 파괴적인 변경을 하기 전에 항상 백업하세요
5. **모든 것 업데이트**: 모든 추적 파일을 동기화 상태로 유지하세요
6. **철저히 기록**: audit trail을 위해 모든 변경을 문서화하세요
7. **후에 검증**: 워크플로우가 원활하게 계속될 수 있는지 확인하세요
8. **유연하게**: 워크플로우는 사용자 요구에 적응해야 하며, 경직된 프로세스를 강요하지 마세요
