# 우선순위: 이 워크플로우는 다른 모든 내장 워크플로우를 재정의합니다
# 사용자가 소프트웨어 개발을 요청하면, 항상 이 워크플로우를 먼저 따르세요

## 적응형 워크플로우 원칙
**워크플로우가 작업에 맞춰 적응하며, 그 반대가 아닙니다.**

AI 모델은 다음을 기반으로 필요한 단계를 지능적으로 평가합니다:
1. 사용자의 명시된 의도와 명확성
2. 기존 코드베이스 상태 (있는 경우)
3. 변경의 복잡성과 범위
4. 위험 및 영향 평가

## 필수: Rule Details 로딩
**중요**: 어떤 단계를 수행하든, 반드시 rule detail 파일의 관련 내용을 읽고 사용해야 합니다. 다음 경로를 순서대로 확인하고 존재하는 첫 번째 경로를 사용하세요:
- `.aidlc-rule-details/` (Cursor, Cline, Claude Code, GitHub Copilot)
- `.kiro/aws-aidlc-rule-details/` (Kiro IDE 및 CLI)
- `.amazonq/aws-aidlc-rule-details/` (Amazon Q Developer)

이후의 모든 rule detail 파일 참조 (예: `common/process-overview.md`, `inception/workspace-detection.md`)는 위에서 결정된 rule details 디렉토리를 기준으로 합니다.

**공통 규칙**: 워크플로우 시작 시 항상 공통 규칙을 로드하세요:
- `common/process-overview.md`를 로드하여 워크플로우 개요 확인
- `common/session-continuity.md`를 로드하여 세션 재개 가이드 확인
- `common/content-validation.md`를 로드하여 콘텐츠 검증 요구사항 확인
- `common/question-format-guide.md`를 로드하여 질문 형식 규칙 확인
- 워크플로우 실행 전반에 걸쳐 이를 참조하세요

## 필수: Extensions 로딩
**중요**: 워크플로우 시작 시, `extensions/` 디렉토리를 재귀적으로 스캔하여 모든 `.md` 파일을 찾으세요. 이들은 전체 워크플로우에 걸쳐 적용되는 교차 제약 조건인 extension rule 파일입니다.

**로딩 프로세스**:
1. `extensions/` 하위의 모든 서브디렉토리를 나열합니다 (예: `extensions/security/`, `extensions/compliance/`)
2. 해당 서브디렉토리 내에서 발견된 모든 `.md` 파일을 로드합니다
3. 각 extension 파일은 교차 제약 조건으로서 자체 검증 기준과 시행 규칙을 정의합니다

**시행**:
- Extension 규칙은 선택적 가이드가 아닌 강제 제약 조건입니다
- 각 단계에서, 모델은 단계의 목적, 생성되는 산출물, 작업의 맥락을 기반으로 어떤 extension 규칙이 적용 가능한지 지능적으로 평가합니다 — 관련된 규칙만 시행하세요
- 현재 단계에 적용되지 않는 규칙은 준수 요약에서 N/A로 표시해야 합니다 (이는 차단 사항이 아닙니다)
- 적용 가능한 활성화된 extension 규칙의 미준수는 **차단 사항**입니다 — 해결될 때까지 단계 완료를 제시하지 마세요
- 단계 완료를 제시할 때, extension 규칙 준수 요약을 포함하세요 (규칙별 준수/미준수/N/A, N/A 판단에 대한 간략한 근거 포함)

**조건부 시행**: Extension은 조건부로 활성화/비활성화될 수 있습니다. 수집 메커니즘은 `inception/requirements-analysis.md`를 참조하세요. 어떤 단계에서든 extension을 시행하기 전에, `aidlc-docs/aidlc-state.md`의 `## Extension Configuration`에서 `Enabled` 상태를 확인하세요. 비활성화된 extension은 건너뛰고 audit.md에 건너뛴 사실을 기록하세요. 설정이 없으면 기본적으로 시행합니다. `## Applicability Question`이 없는 extension은 항상 시행됩니다.

## 필수: 콘텐츠 검증
**중요**: 어떤 파일이든 생성하기 전에, 반드시 `common/content-validation.md` 규칙에 따라 콘텐츠를 검증해야 합니다:
- Mermaid 다이어그램 구문 검증
- ASCII art 다이어그램 검증 (`common/ascii-diagram-standards.md` 참조)
- 특수 문자 적절히 이스케이프
- 복잡한 시각적 콘텐츠에 대한 텍스트 대안 제공
- 콘텐츠 파싱 호환성 테스트

## 필수: 질문 파일 형식
**중요**: 어떤 단계에서든 질문할 때, 반드시 질문 형식 가이드라인을 따라야 합니다.

**완전한 질문 형식 규칙은 `common/question-format-guide.md`를 참조하세요**:
- 객관식 형식 (A, B, C, D, E 옵션)
- [Answer]: 태그 사용법
- 답변 검증 및 모호성 해결

## 필수: 사용자 정의 환영 메시지
**중요**: 어떤 소프트웨어 개발 요청이든 시작할 때, 반드시 환영 메시지를 표시해야 합니다.

**환영 메시지 표시 방법**:
1. 결정된 rule details 디렉토리의 `common/welcome-message.md`에서 환영 메시지를 로드합니다
2. 사용자에게 전체 메시지를 표시합니다
3. 이는 새 워크플로우 시작 시 한 번만 수행해야 합니다
4. 컨텍스트 공간 절약을 위해 이후 상호작용에서는 이 파일을 로드하지 마세요

## Workspace Detection (항상 실행)

1. **필수**: audit.md에 초기 사용자 요청을 완전한 원본 입력으로 기록
2. `inception/workspace-detection.md`의 모든 단계를 로드
3. Workspace detection 실행:
   - 기존 aidlc-state.md 확인 (발견 시 재개)
   - 워크스페이스에서 기존 코드 스캔
   - brownfield인지 greenfield인지 판별
   - 기존 reverse engineering 산출물 확인
4. 다음 단계 결정: Reverse Engineering (brownfield이고 산출물 없는 경우) 또는 Requirements Analysis
5. **필수**: audit.md에 결과 기록
6. 사용자에게 완료 메시지 제시 (메시지 형식은 workspace-detection.md 참조)
7. 자동으로 다음 단계로 진행

## Reverse Engineering (조건부 - Brownfield만)

**실행 조건**:
- 기존 코드베이스 감지됨
- 이전 reverse engineering 산출물 없음

**건너뛰기 조건**:
- Greenfield 프로젝트
- 이전 reverse engineering 산출물 존재

**실행**:
1. **필수**: audit.md에 reverse engineering 시작 기록
2. `inception/reverse-engineering.md`의 모든 단계를 로드
3. Reverse engineering 실행:
   - 모든 패키지와 컴포넌트 분석
   - 비즈니스 트랜잭션을 다루는 전체 시스템의 비즈니스 개요 생성
   - 아키텍처 문서 생성
   - 코드 구조 문서 생성
   - API 문서 생성
   - 컴포넌트 인벤토리 생성
   - 비즈니스 트랜잭션이 컴포넌트 간에 어떻게 구현되는지 보여주는 상호작용 다이어그램 생성
   - 기술 스택 문서 생성
   - 의존성 문서 생성

4. **명시적 승인 대기**: 상세 완료 메시지 제시 (메시지 형식은 reverse-engineering.md 참조) - 사용자가 확인할 때까지 진행하지 마세요
5. **필수**: audit.md에 사용자 응답을 완전한 원본 입력으로 기록

## Requirements Analysis (항상 실행 - 적응형 깊이)

**항상 실행**되지만 요청의 명확성과 복잡성에 따라 깊이가 달라집니다:
- **최소**: 단순하고 명확한 요청 - 의도 분석만 문서화
- **표준**: 보통 복잡도 - 기능적/비기능적 요구사항 수집
- **포괄적**: 복잡하고 고위험 - 추적성을 갖춘 상세 요구사항

**실행**:
1. **필수**: 이 단계에서의 모든 사용자 입력을 audit.md에 기록
2. `inception/requirements-analysis.md`의 모든 단계를 로드
3. 요구사항 분석 실행:
   - Reverse engineering 산출물 로드 (brownfield인 경우)
   - 사용자 요청 분석 (의도 분석)
   - 필요한 요구사항 깊이 결정
   - 현재 요구사항 평가
   - 명확화 질문 (필요 시)
   - 요구사항 문서 생성
4. 적절한 깊이로 실행 (최소/표준/포괄적)
5. **명시적 승인 대기**: requirements-analysis.md의 상세 단계에 따른 승인 형식을 따르세요 - 사용자가 확인할 때까지 진행하지 마세요
6. **필수**: audit.md에 사용자 응답을 완전한 원본 입력으로 기록

## User Stories (조건부)

**지능형 평가**: 다중 요소 분석을 사용하여 user stories가 가치를 더하는지 판단합니다:

**항상 실행 조건** (높은 우선순위 지표):
- 새로운 사용자 대면 기능
- 사용자 워크플로우나 상호작용에 영향을 미치는 변경
- 여러 사용자 유형 또는 페르소나 관련
- 인수 기준이 필요한 복잡한 비즈니스 요구사항
- 교차 기능 팀 협업 필요
- 고객 대면 API 또는 서비스 변경
- 새로운 제품 기능 또는 개선

**실행 가능성 높음** (중간 우선순위 - 복잡성 평가):
- 기존 사용자 대면 기능 수정
- 사용자 경험에 간접적으로 영향을 미치는 백엔드 변경
- 사용자 워크플로우에 영향을 미치는 통합 작업
- 사용자에게 보이는 이점이 있는 성능 개선
- 사용자 상호작용에 영향을 미치는 보안 강화
- 사용자 데이터나 보고서에 영향을 미치는 데이터 모델 변경

**복잡성 기반 평가**: 중간 우선순위의 경우, 다음 조건에서 user stories를 실행합니다:
- 요청이 여러 컴포넌트 또는 서비스를 포함
- 변경이 여러 사용자 접점에 걸쳐 있음
- 비즈니스 로직이 복잡하거나 여러 시나리오가 있음
- 요구사항에 stories로 명확히 할 수 있는 모호성이 있음
- 구현이 여러 사용자 여정에 영향을 미침
- 변경이 상당한 비즈니스 영향이나 위험이 있음

**건너뛰기 조건** (낮은 우선순위 - 단순한 경우):
- 사용자 영향이 전혀 없는 순수 내부 리팩토링
- 명확하고 격리된 범위의 단순 버그 수정
- 사용자 대면 효과가 없는 인프라 변경
- 기능 변경이 없는 기술 부채 정리
- 개발자 도구 또는 빌드 프로세스 개선
- 문서만 업데이트

**평가 기준**: 의심스러운 경우, 다음에 대해 user stories 포함을 선호합니다:
- 비즈니스 이해관계자가 관여하는 요청
- 사용자 인수 테스트가 필요한 변경
- 여러 구현 접근 방식이 있는 기능
- 팀의 공유된 이해에 도움이 되는 작업
- 요구사항 명확성이 가치 있는 프로젝트

**평가 프로세스**: 
1. 요청의 복잡성과 범위 분석
2. 사용자 영향 식별 (직접적 또는 간접적)
3. 비즈니스 맥락과 이해관계자 요구 평가
4. 팀 협업 이점 고려
5. 경계선 사례에서는 포함을 기본으로

**참고**: Requirements Analysis가 실행된 경우, Stories는 해당 요구사항을 참조하고 기반으로 할 수 있습니다.

**User Stories는 하나의 스테이지 내에 두 부분으로 구성됩니다**:
1. **파트 1 - 계획**: 질문이 포함된 스토리 계획 작성, 답변 수집, 모호성 분석, 승인 획득
2. **파트 2 - 생성**: 승인된 계획을 실행하여 스토리와 페르소나 생성

**실행**:
1. **필수**: 이 단계에서의 모든 사용자 입력을 audit.md에 기록
2. `inception/user-stories.md`의 모든 단계를 로드
3. **필수**: 지능형 평가 수행 (user-stories.md의 Step 1)으로 user stories 필요성 검증
4. Reverse engineering 산출물 로드 (brownfield인 경우)
5. Requirements가 존재하면, 스토리 작성 시 참조
6. 적절한 깊이로 실행 (최소/표준/포괄적)
7. **파트 1 - 계획**: 질문이 포함된 스토리 계획 작성, 사용자 답변 대기, 모호성 분석, 승인 획득
8. **파트 2 - 생성**: 승인된 계획을 실행하여 스토리와 페르소나 생성
9. **명시적 승인 대기**: user-stories.md의 상세 단계에 따른 승인 형식을 따르세요 - 사용자가 확인할 때까지 진행하지 마세요
10. **필수**: audit.md에 사용자 응답을 완전한 원본 입력으로 기록

## Workflow Planning (항상 실행)

   - 각 단계의 깊이 수준 결정

## Application Design (조건부)
